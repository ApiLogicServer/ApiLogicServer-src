# Comparison: `rich` vs `skillz`

Both projects live in `~/dev/ApiLogicServer/ApiLogicServer-dev/servers/` and model the
same domain — **roles, skills, certifications, proficiency levels, and progression** —
but were built with fundamentally different approaches:

- **`rich`** — generated by GenAI-Logic / ApiLogicServer
- **`skillz`** — hand-built natively by Rich Damkoeller, also using Claude as the
  coding assistant (same AI, no GenAI-Logic Context Engineering)

That makes this an unusually clean **native-AI-coding vs. GenAI-Logic** comparison:
same domain, same underlying model (Claude), different methodology.

## Extending the thought experiment: an enterprise app with inbound/outbound integration

Neither `rich` nor `skillz` currently has a real B2B/EAI requirement — but it's worth
asking how each approach would extend to one, since most enterprise systems eventually
need to **receive** events/messages from other systems and **publish** events when
their own data changes (order intake from a partner, shipment notifications to a
downstream system, etc.).

**What exists today, concretely:**

- **`rich`** ships with integration scaffolding already in the generated base —
  `integration/kafka/` (producer, consumer, `kafka_subscribe_discovery/` for inbound
  topics), `integration/system/EaiPublishMapper.py` / `EaiSubscribeMapper.py` /
  `RowDictMapper.py` for field-mapping between external payloads and internal models,
  and a documented two-message inbound pattern (save raw payload, then parse/persist
  in a second transaction) with a `/consume_debug/<topic>` endpoint so the flow can be
  tested without a live Kafka broker. Outbound publishing is a single rule:
  `Rule.after_flush_row_event(on_class=Order, calling=kafka_producer.send_row_to_kafka,
  if_condition=lambda row: row.date_shipped is not None, with_args={"topic": "order_shipping"})`.
  None of this is wired up for the roles/skills domain in `rich` today — it's
  unused scaffolding, not an active integration — but the pattern and mapper code
  exist and are documented (`integration/kafka/kafka_readme.md`).
- **`skillz`** has no integration layer at all — no message broker client, no
  inbound webhook/consumer routes, no outbound event publishing. Adding either would
  be a from-scratch build: choosing a broker/queue, writing the producer/consumer
  code, and writing the field-mapping/persistence logic by hand.

**The specific architectural claim worth verifying — "logic guaranteed to apply":**

This is checkable, not just a vendor assertion. LogicBank (the rules engine `rich`
runs on) hooks SQLAlchemy's `before_flush`/`after_flush` session events directly
(`org_git/LogicBank/logic_bank/logic_bank.py`, confirmed). That means **any** code
path that writes through the ORM session — a JSON:API call, a Kafka consumer calling
`session.add()`, an internal script, an admin-app edit — passes through the same
rule engine automatically. A constraint or derivation declared once applies to API
writes *and* B2B message intake *and* future integration points nobody has built yet,
with no separate code path to remember. This is the basis for the "guaranteed to
apply" claim, and it's an architectural property of the ORM-hook design, not
something each new integration has to opt into.

In `skillz`'s architecture, the equivalent guarantee would have to be re-implemented
by hand for each entry point: if a future Kafka consumer or webhook handler is added,
every invariant currently enforced only in `controllers/`/`data/` (e.g., a count
staying in sync, a role-progression rule) would need to be re-applied or refactored
into a shared layer that the new entry point also calls. Nothing in the current
layered design (`routes → controllers → data → pg`) automatically extends business
rules to a code path that doesn't go through those controllers — it's a discipline
the team must maintain, not a property the database/ORM layer enforces.

**On the public claim that GenAI-Logic can generate this integration layer from a
prompt:** the project's marketing makes this claim (e.g., describing inbound/outbound
EAI generation from natural-language prompts on genai-logic.com); I have not
independently verified that claim against this workspace and am reporting it as an
assertion from the vendor/maintainer (who is also the user in this conversation),
not as something I confirmed by testing prompt-driven EAI generation myself. The
`kafka_readme.md` example prompt format (`integration/kafka/kafka_readme.md`) is real
and present in the codebase, which supports that the *capability* exists in the
product — but I did not run a live generation to confirm output quality or
completeness.

## Five further considerations (vendor-supplied, independently checked where possible)

The following points were raised by the GenAI-Logic maintainer (the user in this
conversation) as additional context. Each is evaluated on its own merits below —
some are verifiable directly in this workspace, some rest on architectural
reasoning that follows from facts already confirmed, and one cites an external
experiment I have not independently reproduced.

### 1. Correctness over the dependency graph, not just "cleaner code"

The structural argument is sound and follows from what's already verified: an
N-table system with foreign-key relationships has a combinatorial number of
distinct change paths (insert at any level, update any FK, reassign a child to a
different parent, delete at any level). Procedural code must enumerate each path
explicitly; a missed path is a silent correctness bug (wrong data, no exception)
rather than a crash. Declarative rules, because LogicBank computes the dependency
graph from the rule declarations themselves rather than from someone's
enumeration of cases, cover every path that touches a declared dependency by
construction — including FK-reassignment paths, which are exactly the kind of
case manual enumeration tends to miss.

The cited experiment is documented in detail in this workspace —
`samples/basic_demo_logic_gov/logic/procedural/declarative-vs-procedural-comparison.md`
(read directly, not taken on faith) — with the same 5-rule requirement, the two
bugs (`Order.customer_id` reassignment leaving the old customer's balance stale;
`Item.product_id` reassignment leaving `unit_price` stale), a representative
excerpt of the buggy procedural code (Appendix A), and a Behave test matrix
showing both bugs failing while five happy-path scenarios pass on both
implementations. That's substantially more than a bare vendor claim, and I'm
revising the earlier framing accordingly — it should not be cited as something I
treated as unverifiable.

It is still not something I independently reproduced, and the document is
explicit about the same limits: its own "Scope of the evidence" section states
this was "an informal experiment, reported as observed," and that the *full*
~200-line procedural implementation and test harness — needed for every row of
the results table to be independently reproducible — were not committed to the
repo (only the excerpt in Appendix A is). I did not run the Behave suite myself.
So: methodology and bug mechanism, confirmed by reading the source; the specific
pass/fail results, taken on the document's word, not re-run.

One more distinction worth keeping: the document carries two layers of differing
rigor. The body (through "Scope of the evidence") is hedged and self-limiting, as
above. Appendix B is explicitly marked as AI-generated commentary the author
states he does *not* endorse and did *not* independently investigate — its
stronger claims ("AI cannot fix this," "structural impossibility," "this cannot
be achieved with procedural code") are the assistant's framing, not the author's
verified conclusion, per the document's own note. I'm relying on the body, not
Appendix B, for what follows.

The general mechanism the experiment illustrates is the same one already
confirmed elsewhere in this comparison: `skillz`'s hand-written TypeScript has
exactly this exposure today — every multi-table invariant (e.g.,
`role.skill_count` staying in sync) is a hand-maintained path, correct only to
the extent its author enumerated every way that value can change. The
comparison's original framing ("auto-maintained aggregates reduce stale-data
risk") understated this: it's not just lower risk, it's the only one of the two
architectures where completeness over the dependency graph is structurally
enforced rather than a property of how carefully someone wrote the code.

### 2. Iteration at scale

This follows directly from the same mechanism and is worth stating plainly: at 5
rules, a missed path is unlikely and cheap to find. Enterprise systems accumulate
hundreds of rules over years. Adding or changing one declarative rule lets
LogicBank recompute the dependency graph incrementally — other rules are
unaffected, not regenerated. Changing a business rule in `skillz`'s architecture
means tracing every procedural code path the change could touch, by hand, across
however much of the codebase has grown since the original design. This isn't
primarily a speed difference — it's a safety difference: large procedural
codebases become unsafe to change with confidence without exhaustive re-review,
and "better AI" doesn't fix this, because the limitation is in procedural code's
inability to expose its own dependency graph for inspection, not in who or what
is writing the procedural code.

### 3. Governance artifacts beyond Behave traceability

The comparison's Governance section already credits `rich` with
`docs/requirements/` provenance and Behave traceability. Three more generators
exist in the framework and are worth distinguishing by what's actually been
**produced** versus what's **available**:

- **Ad-libs report — produced and verified.** `rich/docs/requirements/ad-libs.md`
  exists right now, dated 2026-06-16, and lists every assumption made beyond the
  prompt (schema type substitutions, seed-data choices) in a reviewable table.
  This is a real artifact for this project, not just a documented capability.
- **Logic diagrams — available, not yet generated for `rich`.** The generator
  (`docs/training/logic_diagrams/generate_logic_diagram.py`) and reading guide
  exist in the project, but no `logic_flow_*.md`/SVG output has actually been
  produced here yet. The capability (one diagram per requirement, pruned to the
  tables it touches, readable by a non-engineer) is real and documented, but
  describing it as a current artifact of `rich` would overstate what's been run.
- **Governance health check — same status.** `docs/training/health_check.md` and
  `docs/training/governance.md` document the scoring methodology (rule coverage
  per domain table, anti-pattern detection), but no health-check report exists
  in this repo yet either.

The underlying claim — that these reports generate from the rule objects
themselves and so cannot drift from what the code enforces — is consistent with
the architecture already verified (rules are Python objects LogicBank introspects
at startup; a diagram or coverage score derived from that same object graph is
reading the enforced behavior, not a separate description of it that could go
stale). `skillz` has no equivalent: `CHANGE_LOG.md` and `spec/` are real and
well-kept, but they're narrative records a human wrote, not derived from the
enforcement code, so they can drift from it silently.

### 4. Runtime AI as a declared rule

This is a real, documented pattern (`docs/training/probabilistic_logic.md`,
confirmed present) — a rule can call an LLM at transaction time (e.g., "use AI to
pick the optimal supplier considering cost, lead time, and current conditions"),
with the AI's output still passing through the same deterministic rule engine
before commit. It is **not used in `rich` today** — no probabilistic rule exists
in this project's `declare_logic.py`; it's available framework capability, not an
active feature of this comparison's artifact. The architectural point holds
regardless: this is a different category of capability than "AI writes code at
design time" (the frame the rest of this comparison uses for both projects), and
because it's still a declared rule, it inherits the same dependency-tracking and
governance properties as any other rule — that part is not a separate claim, it
follows from the ORM-hook architecture already verified.

### 5. MCP / agentic access inherits the same governance, by the same mechanism

Confirmed in code: any file dropped into `api/api_discovery/` — including
`mcp_discovery.py`, which is present in `rich` — is auto-loaded at server startup
by a generic directory-walk (`api/api_discovery/auto_discovery.py`, confirmed).
No manual registration step is required; this is "on by default" for a generated
project, not something a developer must remember to wire in.

The governance claim that matters here isn't about MCP specifically — it's that
an MCP-discovering agent calling the generated JSON:API still writes through the
same SQLAlchemy session, which is the same chokepoint LogicBank hooks
(`before_flush`/`after_flush`, verified in the integration section above). So a
rule enforced for a human using the Admin App is, by construction, also enforced
for an agent calling the API via MCP — not a second governance system that has to
be separately built and kept in sync, but the identical one. The risk this avoids
is real and worth naming explicitly for an EA evaluating agentic architectures: an
MCP server that just exposes raw database CRUD with no rules layer underneath
gives any agent that finds it unconstrained read/write access to production
data — `rich`'s architecture closes that gap as a side effect of how its rules
engine already works, not as an MCP-specific feature. `skillz` has no MCP
endpoint and, per point 3 above, no equivalent mechanism that would
automatically extend its hand-written validation logic to a new access path
without a developer building it in.

## How an enterprise architect might evaluate these

An EA comparing these two as *candidate approaches* for a real competency-management
system (not just artifacts to inspect) would likely weigh them across several axes:

### 1. Time-to-working-system vs. time-to-production-shape
- **`rich`** gets you a running API + Admin UI + persisted rules in minutes from a
  schema or NL prompt. That's a powerful proof-of-concept and stakeholder demo
  vehicle, but the EA would ask: *is the generated shape (this prototype's SQLite —
  swappable to Postgres/MySQL/Oracle/SQL Server without an architecture change —
  plus a YAML-config admin app and a framework-owned API) the one we want to
  operate in production, or is this a fast way to validate the domain model before
  a "real" build?*
- **`skillz`** took longer to reach a working state but arrived already shaped like
  a deployable product — Postgres, CI/CD, cloud hosting, real auth — because every
  phase was built toward that target from the start.

### 2. Logic correctness and maintainability under change
- **`rich`**'s declarative rules (sums/counts auto-maintained by LogicBank) are the
  framework's strongest architectural claim — and per the dependency-graph
  argument above (see "Five further considerations," #1–#2), it's not just a
  reduced-risk claim: completeness over the change-path graph is structurally
  enforced in this architecture, not a property of how carefully the rules were
  written. An EA evaluating long-lived systems with many derived/aggregated
  fields, or planning for hundreds of rules rather than five, would weigh that
  against the cost of buying into a less-common paradigm.
- **The "smaller hiring pool" cost is reduced, not eliminated, by the Context
  Engineering (CE) that ships with every generated project.** `.github/.copilot-instructions.md`
  plus `docs/training/logic_bank_api.md` (confirmed present in `rich`, and loaded
  into this conversation at session start) give an AI assistant the complete Rule
  API, worked examples, dependency-tracking rules, and anti-patterns — so a
  developer doesn't need prior LogicBank experience to *produce* correct rules.
  They write or review requirements in natural language; the AI, operating under
  that CE, does the NL→`Rule.sum`/`count`/`formula`/`constraint` translation. The
  residual skill required is narrower than "already knows this framework" — it's
  "can read a declared rule and reason about its dependency chain," which is a
  much lower bar.
  - This advantage is conditional, though: it holds only while the CE files
    (`.github/.copilot-instructions.md`, `docs/training/*.md`) are present, current,
    and an AI assistant is in the loop. An EA should treat the CE as part of what's
    being adopted, not assume the low-ramp-up property is free or permanent — see
    "the CE itself is a dependency" under Lock-in, below.
  - It also doesn't fully cover *operating* the runtime: a developer debugging a
    production incident (why didn't this rule fire, why did it fire twice) still
    needs to understand LogicBank's execution model — auto-ordering, auto-reuse,
    re-fire conditions — well enough to reason about it without AI assistance in
    the moment. That's a smaller skill than "learn the whole framework from
    scratch," but it isn't zero.
- **`skillz`**'s logic is conventional imperative TypeScript with explicit
  transactions — no special paradigm to ramp up on at all, CE or no CE, AI or no
  AI; any backend engineer already knows this style. Easier to staff and debug
  with standard tools, but every multi-table invariant (e.g., a count staying
  in sync) is a hand-maintained code path that has to be gotten right and kept
  right as requirements evolve.

### 3. Lock-in and platform dependency
- **`rich`** depends on the ApiLogicServer/LogicBank runtime for its logic and API
  layers — an EA would still flag this as a framework dependency to evaluate
  (community size, longevity, support model, pace of evolution) the same way
  they'd evaluate any platform choice. But unlike a closed low-code/SaaS
  platform, the license is source-available (Apache 2.0 + Commons Clause for
  `ApiLogicServer`; MIT for `genai-logic`) — the EA can read, fork, and patch the
  runtime itself if the vendor relationship ends; the Commons Clause only blocks
  *reselling* the framework as a hosted product, not using it internally.
- **`skillz`** has no framework lock-in beyond mainstream OSS (Express, React,
  `pg`, Vite) — lower platform risk, but all the "plumbing" (API conventions, the
  transaction helper, the junction-reconciliation helper) is bespoke and only as
  good as the team that wrote it. The codebase itself is proprietary to its
  owner (no license file found) — a separate lock-in question (who owns/can
  reuse the code) distinct from the runtime-dependency question above.
- **The CE itself is a dependency worth tracking separately from the runtime.**
  The low platform-familiarity cost described in section 2 is a property of
  LogicBank *plus* the CE files *plus* an AI assistant in the loop — not of
  LogicBank alone. An EA should ask the same lock-in questions of the CE that
  they'd ask of the runtime: is it maintained, does it stay in sync with the
  framework version in use, and does the org's AI-assistant policy (which
  models, which tools) remain compatible with how the CE expects to be invoked.

### 4. Operability and production-readiness
- **`skillz`** already has the artifacts an EA looks for in a production
  candidate: real auth (Google/Firebase), CI gating merges, a deploy pipeline to
  named cloud services, and a migration ledger (`schema_migration`) — these are
  not afterthoughts, they were built in.
- **`rich`** has the deployment mechanics (Docker image, Azure Container Instance
  script — see Deployment row above) but lacks two things `skillz` already has:
  CI/CD (the Azure script is manually run, not push-to-deploy) and activated
  security (one config flag, but the role grants are still generic template
  content and no identity provider is wired up — see Auth row). An EA would
  treat both as gaps to close before production, not flaws in the generator —
  these are exactly the layers GenAI-Logic expects a team to add on top of the
  generated base. The *residual* work to reach where `skillz` already is (CI/CD,
  real domain roles, a real identity provider) is comparable in kind to what
  `skillz` already paid for — it just hasn't been spent yet here.

### 5. Governance, audit, and requirements traceability
- **`rich`** carries built-in requirements traceability — `docs/requirements/`
  (provenance + the ad-libs report, confirmed produced for this project) and
  Behave tests that produce human-readable, requirement-linked execution
  reports. Two further generators (logic diagrams, governance health check —
  see "Five further considerations," #3) are present as framework tooling but
  haven't been run for `rich` yet; an EA evaluating this for a regulated or
  audit-sensitive domain should weigh the *produced* artifact (ad-libs) and the
  *available* ones (diagrams, health check) differently, while noting all of
  them share the same property: they're generated from the rule objects
  LogicBank actually enforces, so they can't silently drift from runtime
  behavior the way a hand-maintained doc can.
- **`skillz`** has a detailed `CHANGE_LOG.md` (good engineering hygiene) and a
  `spec/` folder of pre-build discovery docs, but no equivalent automated
  requirement-to-implementation trace — traceability here is manual/narrative
  rather than structurally enforced.

### 6. Total cost of ownership
- **`rich`**: lower marginal cost to extend the data model and simple
  aggregation/constraint logic (declare a rule vs. write and test procedural
  code); higher cost if the team ever needs to step outside what the framework
  models well, or if the framework's pace of evolution doesn't match the
  project's needs.
- **`skillz`**: each individual change is a normal, well-scoped engineering task
  — the v0.12–v0.20 entries in `CHANGE_LOG.md` show real, predictable,
  incremental cost, with no special tooling payoff but also no special tooling
  risk. What the per-feature cost doesn't include is the compounding tail risk
  named in section 2 above: nothing in this architecture lets anyone — the
  original team, a new hire, an auditor, an acquirer — confirm from outside the
  code that a given invariant (e.g., `role.skill_count` staying in sync) still
  holds after N "predictable" changes, short of an expert manually re-reading
  the relevant code paths. That verification cost is real, grows with the
  codebase, and is not priced into any single feature's cost — it's deferred,
  not absent.
- **The asymmetry that matters for TCO is *when* the cost is paid, not just how
  much.** `rich`'s costs (platform dependency, paradigm unfamiliarity, the CE
  dependency noted in section 3 below) are bounded and front-loaded — an EA can
  audit them on day one by reading the CE and the license. `skillz`'s
  correctness-drift risk is unbounded and back-loaded — it compounds with every
  change and is not discoverable on a fixed schedule; it surfaces when a bug
  manifests in production or when someone goes looking, and "going looking"
  becomes the more expensive and less tractable option as the codebase grows.
  This is a difference in risk *shape*, not just risk size, and a TCO estimate
  that treats them as comparable line items understates `skillz`'s long-run
  number.

### Why governance is not just item 5 of 6

The "Five further considerations" above are listed as five separate points, but
they aren't peers. Governance — rules being inspectable as data, not just
correctly executed — is the property that makes the long-term validity of the
correctness argument (section 2), the lock-in argument (section 3), and the TCO
argument above actually hold up *over time*, rather than only at first build.
A system whose rules are declared as data can, in principle, be audited at
scale without re-reading procedural code line by line; a system whose
invariants live only in hand-written control flow cannot, regardless of how
careful the team is. The cited source document's own framing of this — "the
single-project result is the brick; governance across a portfolio is the
wall" — is a claim about a different system at a different scale (960 rules
across 100 tables) than either `rich` or `skillz`, so it should be read as
illustrating the mechanism, not as evidence already demonstrated by either
project here. In `rich` specifically, that governance infrastructure is mostly
*available* rather than *exercised* today: per section 3 above, the ad-libs
report is produced and verified for this project, but the logic-diagram and
health-check generators have not yet been run against it. So the accurate
claim is narrower than "governance is proven at scale" — it's that `rich`'s
architecture makes portfolio-scale auditability *possible by construction*,
while `skillz`'s architecture does not make it possible at all, independent of
how disciplined the team is.

### Net framing for an EA
This pair is a useful natural experiment specifically *because* the AI (Claude)
is held constant and the variable is the methodology: GenAI-Logic's Context
Engineering and declarative-rules runtime vs. native, framework-free AI-assisted
coding. The architect's decision is not "which artifact is better" but a choice
between two different shapes of risk: `rich`'s risk (platform dependency,
paradigm unfamiliarity, a CE dependency to track) is bounded and visible on
inspection; `skillz`'s risk (hand-maintained invariants whose correctness
cannot be confirmed from outside the code, and which compounds silently as the
system grows) is unbounded and not discoverable on any fixed schedule. Both are
